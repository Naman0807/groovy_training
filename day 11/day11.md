## 2. Embeddings Theory Notes

### What Are Embeddings?

**Embeddings** are dense vector representations of text — arrays of floating-point numbers (e.g., 768 dimensions for `text-embedding-004`) that capture semantic meaning.

```
"cat"  → [0.023, -0.045, 0.112, ..., 0.008]  (1536 numbers)
"kitten" → [0.021, -0.042, 0.109, ..., 0.007]  (similar vector)
"car"  → [-0.067, 0.031, -0.089, ..., 0.054]  (different vector)
```

### Key Properties

1. **Semantic proximity** — Similar texts produce similar vectors
2. **Fixed dimensionality** — Always the same length regardless of input length
3. **Dense** — Most dimensions are non-zero (vs sparse one-hot encoding)
4. **Learned** — Generated by a neural network trained on massive text corpora

### Cosine Similarity

The most common metric for comparing embeddings:

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)
```

- **Range:** -1 to 1 (in practice, 0 to 1 for text embeddings)
- **1.0** — identical direction (same meaning)
- **0.7–0.9** — very similar (paraphrase, same topic)
- **0.4–0.7** — somewhat related (same domain)
- **<0.3** — largely unrelated

**Why cosine?** Because it's length-normalized — we care about direction (meaning), not magnitude (length).

### Embedding Models Comparison

| Model | Dimensions | Price | Max tokens | Quality |
| :---- | :--------- | :---- | :--------- | :------ |
| text-embedding-004 | 768 | $0.0375/M chars | 2048 | Best |
| models/embedding-001 | 768 | Free tier avail. | 2048 | Good |
| text-embedding-3-small | 1536 | $0.02/M tok | 8191 | Good |
| text-embedding-3-large | 3072 | $0.13/M tok | 8191 | Best |

### Practical Notes

- **Dimensionality:** text-embedding-004 outputs 768 dimensions (smaller = cheaper storage, faster search)
- **Batching:** Always batch embedding API calls (Google supports batch embedding via `batchEmbedContents`)
- **Normalization:** Google embeddings are normalized — cosine sim = dot product
- **Caching:** Cache embeddings for repeat queries (hash the text, store vector)

## 3. ChromaDB Setup

### Option A: Local ChromaDB (Python)

```bash
pip install chromadb
chroma run --path ./chroma_data
```

**Node.js client:**
```bash
npm install chromadb
```

```javascript
const { ChromaClient } = require("chromadb");
const client = new ChromaClient({ path: "http://localhost:8000" });
const collection = await client.getOrCreateCollection({ name: "documents" });
await collection.add({ ids: ["chunk-0"], embeddings: [[...]], metadatas: [{ text: "..." }] });
```

### Option B: JSON File Store (Chosen for Today)

We use a **JSON file-based vector store** (`backend/store/vectors.json`) because:

- Zero infrastructure — no Python, no Docker
- Persists across server restarts
- Simple cosine similarity search in-memory
- Easy to inspect and debug
- Upgrade path to ChromaDB when needed (same API surface)

### Option C: Supabase Vector (pgvector)

```sql
-- Enable pgvector extension
CREATE EXTENSION vector;

-- Create table
CREATE TABLE documents (
  id SERIAL PRIMARY KEY,
  content TEXT,
  embedding VECTOR(1536)
);

-- Search
SELECT content, 1 - (embedding <=> '[0.023,...]') AS similarity
FROM documents
ORDER BY embedding <=> '[0.023,...]'
LIMIT 3;
```

### Our Vector Store Interface

| Method | Description |
| :----- | :---------- |
| `addDocument(info, chunks, embeddings)` | Store document + its chunk vectors |
| `search(queryEmbedding, topK)` | Cosine similarity search → top K chunks |
| `getStats()` | Document count, chunk count |
| `clearStore()` | Reset the store |

### Performance Notes

- **JSON store:** ~1ms search for 100 chunks, ~10ms for 1000 chunks
- **ChromaDB:** ~2ms for 100K vectors (with indexing)
- **pgvector:** ~5ms for 1M vectors (with IVFFlat index)

## 4. RAG Upgrade Approach

### From Day 10 to Day 11

```
Day 10:                   Day 11:
┌────────────────┐        ┌───────────────────────┐
│ Full doc sent  │        │ 1. Chunk document      │
│ with each      │        │ 2. Embed each chunk    │
│ query          │        │ 3. Store in vector DB  │
│                │        │ 4. Embed question      │
│ Cost: ~$0.077  │        │ 5. Cosine search → top3│
│ per query      │        │ 6. Only top-3 sent to  │
│                │        │    Gemini 2.5 Flash    │
│                │        │ 7. Cost: ~$0.0001/query│
└────────────────┘        └───────────────────────┘
```

### Architecture Changes

| Component | Day 10 | Day 11 |
| :-------- | :----- | :----- |
| `POST /api/upload` | Just extract text | Extract → chunk → embed → store |
| `POST /api/ask` | Send full doc + question | Embed question → search → send top-3 chunks |
| `backend/` | 2 files (server, upload) | 5 files (server, chunking, embeddings, vectorstore, upload) |
| Frontend | Basic Q&A | Adds chunk visualization + relevance scores |

### Chunking Strategy

```
Document: "The quick brown fox jumps over the lazy dog..."
                                                         
Chunk 1 (chars 0-500):  "The quick brown fox..."        
Chunk 2 (chars 450-950): "jumps over the lazy dog..."   
                         └─── 50 char overlap ───┘
```

- **Chunk size:** 500 characters (≈125 tokens)
- **Overlap:** 50 characters (prevents context split at boundaries)
- **Why 500?** Balances semantic coherence with granularity — each chunk is ~2-3 paragraphs

### RAG Query Flow (Detailed)

```
User: "What does the document say about caching?"
  │
  ├─ 1. Generate embedding for question
  │     → vect = [0.023, -0.045, ..., 0.112]
  │
  ├─ 2. Cosine similarity against all chunk vectors
  │     → scores: [0.89, 0.72, 0.45, 0.32, 0.21, ...]
  │
  ├─ 3. Take top 3 chunks (≥0.4 similarity)
  │     → chunk-14 (0.89): "Caching improves performance by..."
  │     → chunk-15 (0.72): "There are two types of caches..."
  │     → chunk-16 (0.45): "Cache invalidation is hard..."
  │
  ├─ 4. Construct prompt with context
  │     System: "Answer based on these excerpts..."
  │     Context: [Chunk 14] (relevance: 89.0%) ...
  │               [Chunk 15] (relevance: 72.0%) ...
  │               [Chunk 16] (relevance: 45.0%) ...
  │     User: "What does the document say about caching?"
  │
  ├─ 5. Send to Gemini 2.5 Flash
  │
  └─ 6. Parse JSON response → answer + citations + cost
```

### Citation Format (v2)

From Day 10's `{ page, text }` to Day 11's `{ chunk, source, text }`:

```json
{
  "citations": [
    { "chunk": 14, "source": "my-doc.txt", "text": "Caching improves performance..." }
  ]
}
```

## 5. Cost Comparison: Full-Doc vs RAG

### Simplified Table

| Item | Full-Doc (Day 10, Claude) | RAG (Day 11, Gemini) |
| :--- | :------------------------- | :-------------------- |
| **Per query (100KB doc)** | ~$0.077 | ~$0.0001 |
| **1,000 queries** | ~$77.00 | ~$0.10 |
| **Monthly, 10 docs** | ~$77.00 | ~$0.11 |
| **Context sent** | 25,000 tokens | ~300 tokens |
| **Citation source** | Page markers | Vector similarity |

### Breakdown (per query, 25K-token doc)

```
Full-Doc (Claude 3.5 Sonnet):
  Input:  25,000 tok × $3.00/M  = $0.0750
  Output:   150 tok × $15.00/M  = $0.0023
  Total:                        = $0.0773

RAG (Gemini 2.5 Flash):
  Embed question:  200 chars × $0.0375/M  = $0.000008
  Vector search:                            = $0.000000  (local computation)
  Input:         300 tok × $0.10/M         = $0.000030
  Output:        150 tok × $0.40/M         = $0.000060
  Total:                                    = $0.000098

Savings: 99.9%
```

### One-Time Ingestion Cost

```
Chunking 100KB doc → 52 chunks × 500 chars
Embedding 52 chunks → ~25K chars × $0.0375/M = $0.00094

Compare: That's cheaper than 1/80th of a single full-doc query.
```

### When RAG Wins

| Scenario | Full-Doc (Claude) | RAG (Gemini) | Winner |
| :------- | :---------------- | :------------ | :----- |
| 1 query, 10KB doc | $0.008 | $0.00009 | **RAG** |
| 100 queries, 100KB doc | $7.70 | $0.01 | **RAG** |
| 10K queries, 1MB doc | Not possible | $1.00 | **RAG** |
| Real-time (low latency) | ~3s | ~1.5s | **RAG** |

See [`rag-doc-qa/cost_comparison.md`](./rag-doc-qa/cost_comparison.md) for full analysis.

## 7. Prompts Used

### RAG System Prompt

```
You are a document Q&A assistant. You are given the most relevant excerpts
from a document retrieved by a vector search.

Your job:
1. Answer the user's question based ONLY on the provided excerpts.
2. For each statement, cite the specific chunk number and source filename.
3. If the excerpts don't contain enough information, say so clearly.
4. Keep answers concise but complete.

Return your answer in this JSON format:
{
  "answer": "Your answer text here...",
  "citations": [
    { "chunk": 1, "source": "filename.txt", "text": "Excerpt from source..." },
    { "chunk": 3, "source": "filename.txt", "text": "Another excerpt..." }
  ]
}
```

### RAG User Prompt Template

```
Relevant document excerpts:
---
[Chunk 14] (relevance: 89.0%) [Source: my-doc.txt]
Caching improves performance by storing frequently accessed data...

[Chunk 15] (relevance: 72.0%) [Source: my-doc.txt]
There are two types of caches: local and distributed...

[Chunk 3] (relevance: 45.0%) [Source: my-doc.txt]
Cache invalidation is one of the hardest problems...
---

Question: What does the document say about caching?

Answer based on the excerpts above. Cite chunk numbers and sources.
```

### Embedding Generation (Not a prompt, but the API call shape)

```javascript
const { GoogleGenerativeAI } = require("@google/generative-ai");
const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);
const model = genAI.getGenerativeModel({ model: "text-embedding-004" });
const result = await model.batchEmbedContents({
  requests: [
    { content: { parts: [{ text: "Chunk text here..." }] } },
    { content: { parts: [{ text: "Another chunk..." }] } },
  ],
});
// result.embeddings[0].values → [0.023, -0.045, ..., 0.008]
```

### Prompt Engineering Notes

- **Chunk relevance in context** — Adding `(relevance: 89.0%)` helps the LLM weight the sources correctly
- **Source filename** — Including the source prevents hallucination about which document a fact came from
- **JSON format** — Same technique as Day 10, adapted for chunk-based citations
- **"If not enough info" guard** — Critical for RAG: the model must recognize when relevant chunks weren't retrieved
- **No page numbers** — RAG doesn't use page markers; citations reference chunk IDs instead

### Prompt Version History

| Version | Changes | Result |
| :------ | :------ | :----- |
| v1 | Basic "answer from excerpts" | Model ignored relevance scores |
| v2 | Added relevance % in context | Model correctly weighted sources |
| v3 | Added "If not enough info" guard | Zero false claims in test set |
| v4 | Added JSON format + chunk citations | Consistent parseable output |
| v5 (final) | Added source filename to each chunk | Accurate multi-doc citation |

> **Biggest takeaway:** RAG is not just a cost optimization — it's a fundamentally different architecture. Day 10 treated the LLM as a smart reader that could scan the whole document. The 96% cost saving is a side effect of this architectural shift, not the goal itself.
